SECTION 2

解決する課題

このアプリが解こうとしているのは
「KGIを立てても、今週のKDIに落とし込めず、第二領域が後回しになり続ける」こと

01

KGIが宙に浮く

「英語力を上げたい」というKGIはあっても、今週の行動（KDI）に落とし込まれていないと何も進まない。KGI → KPI → KDI の3階層で、目標を今日の行動まで分解する。

02

KPIとKDIが混在する

「計測する指標（KPI）」と「実際にやること（KDI）」が混在すると、何を管理しているのか曖昧になる。KPIペインとKDIペインを分けることで、役割を明確にする。

03

振り返りが形骸化する

Check/Actionを別ツールで行うと、KDIの状態と連動せず形骸化しやすい。週次レビューをKDIと同じ画面の第4ペインに置き、「何がDOで何がTODOか」を見ながら振り返る。

04

進捗を仲間と共有しにくい

グループセッションでKGI〜KDIの進捗を見せるには準備が必要だった。URLを1つ共有するだけで、ログイン不要でボード全体を閲覧できる読み取り専用ビューを用意した。

工夫したポイント

特にこだわったのは
「KGI→KPI→KDIの階層を、機能を増やさずに体感できるようにすること」

📊

工夫① KDIの達成率がKPI→KGIへ自動で伝播する

KDI（行動）の達成率（0〜100%）を入力するだけで、KPI（中間指標）の達成率・KGI（最終目標）の達成率が自動で更新される。「目標は今何%か」を毎回手計算する手間がゼロになる。

達成率を入力

KPI（中間指標）
平均を自動計算
例：シャドーイング継続率 → 73%（自動）
KGI（最終目標）
平均を自動計算
例：英語スピーキング → 67%（自動）
効果: KGIの達成率バーが動くのは、KDIを進めたときだけ。「行動すれば目標が前進する」という感覚が画面に表れる。

SECTION 4

苦戦したポイント

いちばん時間がかかったのは
「KGI・KPI・KDIの違いを画面の構造に落とし込むこと」

苦戦 ①

KPIとKDIをどう分けるか — 混在すると管理の意味が薄れる

最初は「課題（Issue）」と「タスク（Task）」という一般的な言葉で設計していた。しかし「計測する指標（KPI）」と「実際にやる行動（KDI）」は性質が異なる。KPIは「週3回レッスンを受けた率」のような計測値で、KDIは「レッスンの予約を入れる」という具体的行動だ。この2つを混在させると「何を管理しているのか」が曖昧になる。

結論: KPIペインとKDIペインを明示的に分け、KGI+課題を1ペインにまとめることで、4ペインが「目標→指標→行動→振り返り」という鬼速PDCAの思考ステップそのものになった。
苦戦 ②

TODO/DO/DONEの3ステータスを、AIが「未着手/進行中/完了」に戻してしまう

AIに実装を依頼すると、ステータスラベルを日本語の「未着手／進行中／完了」で生成してくる。KDIの文脈では「TODO（まだ手をつけていない）」「DO（今週取り組んでいる）」「DONE（完了した）」という英語3語のほうが、行動の勢いとともに伝わりやすい。しかし何度指示してもAIは汎用的な日本語ラベルに戻す傾向があった。

結論: ステータスの名前と意味を「KDIは行動なのでTODO/DO/DONEにする」と明示的に仕様書（グリルの決定事項）に書いてから実装した。仕様を言語化しておくことが、AIを正しく動かすための前提だと学んだ。
苦戦 ③

「どこまで作るか」の線引き — 5/16のグループセッションに間に合わせる

グリル（壁打ち）で要件を詰めると、「KDIをKPIに自動連携する」「ドラッグ&ドロップで並び替え」「KGIごとに達成期日のカウントダウン」など、追加したい機能がどんどん出てきた。しかしセッション日（5/16）まで4日しかなく、すべては作れない。

結論: 「今回はしない」リストをグリルの中で明示的に決めてから実装に入った。機能を足す議論より、削る議論のほうが難しく時間がかかった。これが今回いちばんの学びだった。

SECTION 5

何を・どこに保存するか

Webアプリの「保存」は1か所ではない。
データ・コード・シークレットの3つは、それぞれ別の場所に入れるのが原則です。

BUCKET 1
🗄️ Neon DB
ユーザーごとに違う・時間とともに変わるもの
KGI（最終目標）
ユーザーが入力・更新するデータ
課題・KPI・KDI
KGIに紐づいて増減するデータ
週次レビュー・Actionアイテム
週ごとに新しく作られるデータ
共有トークン（share_token）
ユーザーごとに自動生成される一意の値
BUCKET 2
📁 GitHub リポジトリ
誰がクローンしても同じ動作をする設計図
TypeScript / React コード
画面・ロジック・コンポーネントの実装
SQLスキーマ（001_initial.sql）
「どんなテーブルを作るか」の設計書。データ本体ではない
package.json / 設定ファイル
どのライブラリをどのバージョンで使うかの定義
.env.local.example
「どんな環境変数が必要か」の雛形（値は書かない）
BUCKET 3
🔐 シークレット
漏洩すると第三者にDBを操作される情報
.env.local（ローカル）
.gitignoreで除外済み。自分のPCにだけ存在する
Vercel Environment Variables
本番サーバーにだけ安全に渡す。GitHubには入れない
⚠️ GitHubに絶対に入れないもの
Neon接続文字列（DATABASE_URL）・NextAuth シークレット。一度公開すると即座に悪用されるリスク
3つに分ける判断ルール
