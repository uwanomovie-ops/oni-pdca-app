2
SECTION 2

解決する課題

いちばん困っていたのは、第二領域のやることが見えなくなって「やってなかった」ことに気づけないこと。
このアプリは、やることを1画面に集め、進捗と振り返りまで続けられるようにした。

01

やることの一元管理と可視化

目標や今週の行動がバラバラの場所にあると、すぐ見えなくなる。「やるつもりだったのに、やってなかった」が起きる。KGI・KPI・KDIを4ペインの1画面にまとめ、開けば全部見える状態にした。

02

振り返りが形骸化する

振り返りを別ツールでやると、KDIの進みとつながらず形だけになる。Check / ActionをKDIと同じ画面の第4ペインに置き、DOとTODOを見ながら週次で振り返れるようにした。

03

納期遅れに気づくのが遅い

期限が近いのに進捗が低いKDIは「注意」（黄）、期限を過ぎたものは「緊急」（赤）と表示。順調なものは何も出さない。色を見るだけで「このままだとまずい」と分かるようにした。

SECTION 3

工夫したポイント

機能を増やさず、「見える・続く・危ないと分かる」を4つに絞った。

📊

工夫① 行動の達成率が、上の目標バーに自動反映される

KDI（今週やること）の達成率を入れると、KPI・KGIのバーも自動で動く。手計算なしで「目標は今何%か」が分かる。

🚦

工夫② 納期と進捗が色で一目わかる

KDI・KPIのカード左端に色のラインを出す。納期を過ぎた・もう間に合わないものは赤、「このままだと納期遅れちゃう」進捗は黄。順調なものは何も出さない。開いた瞬間に「どこを急ぐか」が分かる。

黄 — 注意
期限が近いのに達成率が低い（このままだと遅れそう）
赤 — 緊急
納期を過ぎている、または間に合わない

あわせて、KDIは TODO / DO / DONE の3段階で管理。バッジをクリックするだけで切り替え、DONEにすると達成率100%になる。

TODO
→
DO
→
DONE
📅

工夫③ 振り返りも同じ画面・週ごとに残る

第4ペインで週次の Check / Action を書き、矢印で過去の週にも戻れる。別ノートを開かずにPDCAが1画面で回る。

🤖

工夫④ AIが Plan と振り返りを手伝う（発展課題）

KGIから KPI/KDI の草案を提案（AI分解）。週次の振り返りから Adjust 案とコーチ所見を出す（AI週次コーチ）。どちらもプレビューしてから採用し、DBに残す。

SECTION 4

苦戦したポイント

いちばん時間がかかったのは
「KGI・KPI・KDIの違いを画面の構造に落とし込むこと」

苦戦 ①

KPIとKDIをどう分けるか — 混在すると管理の意味が薄れる

最初は「課題（Issue）」と「タスク（Task）」という一般的な言葉で設計していた。しかし「計測する指標（KPI）」と「実際にやる行動（KDI）」は性質が異なる。KPIは「BtoBサポート10件受注」のような中間目標で、KDIは「見込み客10社にアプローチする」という具体的行動だ。この2つを混在させると「何を管理しているのか」が曖昧になる。

結論: KPIペインとKDIペインを明示的に分け、KGI+課題を1ペインにまとめることで、4ペインが「目標→指標→行動→振り返り」という鬼速PDCAの思考ステップそのものになった。
苦戦 ②

ステータス名が、AIのたびに別の言葉に変わる

KDIの状態は「TODO / DO / DONE」で統一したかった。ところが AI に実装を頼むと「未着手 / 進行中 / 完了」のような別ラベルに書き換えられてしまう。何度直しても同じことが起きた。

結論: 先に「この3語を使う」と決めてメモに残してから作る必要があった。細かいルールは、口頭より書き留めたほうが確実。
苦戦 ③

「どこまで作るか」の線引き — 5/16のグループセッションに間に合わせる

グリル（壁打ち）で要件を詰めると、「KDIをKPIに自動連携する」「ドラッグ&ドロップで並び替え」「KGIごとに達成期日のカウントダウン」など、追加したい機能がどんどん出てきた。しかしセッション日（5/16）まで4日しかなく、すべては作れない。

結論: 「今回はしない」リストをグリルの中で明示的に決めてから実装に入った。機能を足す議論より、削る議論のほうが難しく時間がかかった。これが今回いちばんの学びだった。
5
SECTION 5

保持できるようにしたデータ

課題の問いは「リロードしても消したくないものは何か」です。
このアプリでは、4ペイン上の入力・更新がすべて Neon DB に保存され、ブラウザを閉じても残るようにしました。

保存先と、それを選んだ理由

保存先 ①
Neon PostgreSQL
選んだ理由：Vercel とネイティブ連携でき、サーバーレスで運用コストが低い。KGI→課題→KDI の親子関係を表で設計しやすい。課題の「データベースの箱」要件に合う。
入れるもの：goals / issues / tasks / reviews / action_items / workspaces
アクセス：Next.js API Routes 経由（DATABASE_URL）
保存先 ②
GitHub リポジトリ
選んだ理由：画面・API・型定義は「設計図」であり、ユーザーごとに変わらない。バージョン管理して再現性を確保する。DB の中身そのものは入れない。
入れるもの：React/Next.js コード、db/schema.sql、package.json、.env.example
URL：github.com/uwanomovie-ops/oni-pdca-app
保存先 ③
Vercel 環境変数
選んだ理由：DB 接続文字列や Gemini API キーは漏洩すると第三者に操作される。GitHub に載せず、本番サーバーにだけ渡す必要がある。
入れるもの：DATABASE_URL、GEMINI_API_KEY 等
ローカルは .env.local（.gitignore 済み）

判断ルール（3つに分ける）

DB（Neon）に入れる
ユーザーごとに違う
時間とともに変わる
GitHub に入れる
誰がクローンしても
同じ動作をする
シークレットに置く
漏洩すると
悪用される

※ AI の提案結果そのものは永続化せず、採用した KDI・振り返り・コーチ所見を DB に残す設計にした（再生成は都度 API 呼び出し）。

ADDITIONAL

おまけ — 2ヶ月目の課題とのつながり

4ヶ月目の本題はこの PDCA アプリですが、2ヶ月目「面倒な報告をアプリ化する」で作った仕組みも、同じ週次運用のなかで動いています。

📅 Google Calendar（計画） → ⏱ LINE 実動記録 → ⚡ 鬼速PDCA（振り返り）

カレンダーに入れた予定の稼働時間に対し、作業の切り替え時に LINE ボタンで実動時間を記録する（2ヶ月目課題の GAS + LINE Bot）。面倒な報告を2タップにしたものです。

週次の Check / Adjust はこの PDCA アプリで行い、「何時間使ったか」は LINE、「何を達成したか」は PDCA — 役割を分けて同じ週を振り返っています。

📅 Google Calendar — [BtoB] 等の接頭辞付きで、週の計画（稼働予定）を管理
⏱ 実動 — 2タップで記録
📊 サマリー — 今週の実動合計
💰 予算残 — 計画と実動の差分
